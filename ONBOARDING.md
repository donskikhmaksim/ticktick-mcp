# Онбординг: своя копия сервера для нового человека

Каждый человек получает **отдельный** инстанс на Railway со **своими** токенами
TickTick — тогда у него управляется его личный аккаунт, а не чужой. Одна копия =
один человек. Пройди шаги ниже для каждого нового пользователя.

> Кратко: 1) получить два токена TickTick этого человека, 2) поднять отдельный
> Railway-сервис из этого репозитория, 3) вписать токены в переменные,
> 4) сгенерировать домен, 5) добавить ссылку коннектором в его Claude.

---

## Что понадобится от человека

- Аккаунт TickTick (его личный).
- Залогиненный ticktick.com в браузере на компьютере (для `t`-куки).
- Аккаунт Claude с возможностью добавлять custom-коннекторы (Pro/Team/Enterprise).

---

## Шаг 1. Токен официального API (`TICKTICK_ACCESS_TOKEN`)

Этот токен даёт базовые функции (проекты, задачи, создание/изменение/завершение).

1. Developer-приложение TickTick регистрируется **один раз** (твоё), его
   `CLIENT_ID` / `CLIENT_SECRET` переиспользуются для всех — на
   <https://developer.ticktick.com> → Manage Apps. В Redirect URI укажи
   `http://localhost:8000/callback`.
2. На компьютере (твоём или человека) получи токен ИМЕННО ЭТОГО человека —
   он должен залогиниться в открывшемся окне под своим TickTick:
   ```bash
   git clone https://github.com/donskikhmaksim/ticktick-mcp
   cd ticktick-mcp
   uv venv --python 3.12 && uv pip install -r requirements.txt
   cp .env.template .env        # впиши CLIENT_ID и CLIENT_SECRET
   uv run -m ticktick_mcp.cli auth
   ```
3. В браузере человек жмёт Allow под своим аккаунтом. Токен запишется в `.env`
   как `TICKTICK_ACCESS_TOKEN` — скопируй его значение, понадобится на шаге 3.

> Токен официального API долгоживущий (refresh-токен тоже сохраняется), но если
> протухнет — повтори `auth` под этим человеком.

---

## Шаг 2. Токен v2 — кука `t` (`TICKTICK_V2_TOKEN`)

Даёт «продвинутое»: завершённые задачи, теги, корзину, перенос между списками,
activity-историю, `get_changes`. **Получается только вручную** — TickTick закрыл
вход по паролю капчей, автоматизировать нельзя.

1. Человек открывает <https://ticktick.com> в браузере и логинится.
2. DevTools (F12) → вкладка **Application** (Chrome) / **Storage** (Firefox) →
   **Cookies** → `https://ticktick.com`.
3. Найти cookie с именем **`t`**, скопировать его **Value** (длинная строка).
4. Это значение → `TICKTICK_V2_TOKEN` на шаге 3.

> Без этой куки сервер всё равно работает, но без v2-функций. Кука долгоживущая,
> но однажды протухнет — тогда просто повторить этот шаг и обновить переменную.

---

## Шаг 3. Поднять отдельный Railway-сервис

1. Railway → **New Project → Deploy from GitHub repo** → выбрать
   `donskikhmaksim/ticktick-mcp`. Соберётся из `Dockerfile` автоматически.
2. Railway → **Variables**, задать:

   | Переменная | Значение |
   |---|---|
   | `MCP_TRANSPORT` | `streamable-http` |
   | `MCP_SECRET` | свой длинный секрет: `openssl rand -hex 24` (у каждого человека УНИКАЛЬНЫЙ) |
   | `TICKTICK_ACCESS_TOKEN` | из шага 1 (его токен) |
   | `TICKTICK_V2_TOKEN` | из шага 2 (его кука `t`) |
   | `TICKTICK_CLIENT_ID` | общий developer-app id |
   | `TICKTICK_CLIENT_SECRET` | общий developer-app secret |
   | `USER_TIMEZONE` | его таймзона, напр. `Europe/Moscow` или `America/Los_Angeles` |

   `PORT` Railway проставит сам — не трогать.
3. Railway → **Settings → Networking → Generate Domain**.
4. Итоговая ссылка: `https://<его-сервис>.up.railway.app/mcp/<его-MCP_SECRET>`.

---

## Шаг 4. Подключить в Claude человека

В приложении Claude человека: **Settings → Connectors → Add custom connector** →
вставить полную ссылку из шага 3 (включая `/mcp/<секрет>`).

Готово — у него управляется его личный TickTick.

---

## Безопасность

- Ссылка с секретом = полный доступ к TickTick этого человека. Передавать
  только ему, по приватному каналу. Ротация — сменить `MCP_SECRET` в Railway.
- Токены одного человека НЕ переиспользовать для другого — каждому свои, иначе
  все будут править один и тот же аккаунт.
- `CLIENT_ID` / `CLIENT_SECRET` общие — это нормально, они идентифицируют
  developer-приложение, а не пользователя; пользователя определяет access-токен.

## Чеклист на каждого нового человека

- [ ] Получен его `TICKTICK_ACCESS_TOKEN` (шаг 1, логин под ним)
- [ ] Получена его кука `t` → `TICKTICK_V2_TOKEN` (шаг 2)
- [ ] Создан отдельный Railway-сервис из репо
- [ ] Заданы все переменные, `MCP_SECRET` уникальный
- [ ] Сгенерирован домен
- [ ] Ссылка добавлена коннектором в его Claude
- [ ] Проверка: попросить Claude «покажи мои проекты в TickTick»
